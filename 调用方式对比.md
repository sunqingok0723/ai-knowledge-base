# Sub-Agent 调用方式对比

## 概述

在 Claude Code 中，有两种委派 Sub-Agent 的方式：

1. **@mention 方式**：在主对话中直接提及 Agent 角色
2. **Agent Tool 方式**：使用 Agent 工具启动独立的子会话

本文档对比这两种方式的区别，并提供使用建议。

---

## 核心区别对比

| 维度 | @mention | Agent Tool |
|------|----------|------------|
| **上下文** | 共享主对话上下文，可见历史记录 | 独立上下文，从空白开始，无历史 |
| **并行性** | 串行执行，一次只能运行一个 | 可并行执行多个独立任务 |
| **工具访问** | 全部工具可用 | 取决于 subagent_type（如 Explore 只读） |
| **内存访问** | 可访问主对话记忆和项目状态 | 独立记忆，无法访问主对话状态 |
| **成本** | 占用主上下文窗口 | 独立计费，不占主上下文 |
| **结果可见性** | 执行过程实时可见 | 需要等待完成后查看结果 |
| **适用任务** | 简单、快速、需要上下文的任务 | 复杂、多步骤、可独立处理的任务 |

---

## 使用决策树

```
需要执行子任务
    │
    ├─ 需要访问对话历史或当前状态？
    │   └─ 是 → 使用 @mention
    │       例如："@collector 采集今天的数据"
    │             （我已知道今天的日期和之前的状态）
    │
    ├─ 需要并行执行多个独立任务？
    │   └─ 是 → 使用 Agent Tool
    │       例如：同时搜索不同代码库的函数定义
    │
    ├─ 任务复杂且包含多个步骤？
    │   └─ 是 → 使用 Agent Tool (general-purpose)
    │       例如：重构整个认证系统
    │
    └─ 只是简单的查询或阅读？
        └─ 是 → 直接使用 Read/Grep 工具，不需要 Sub-Agent
```

## 示例对比

### @mention 适用场景

**场景 1：简单任务**
```markdown
你: @collector 采集本周 AI 领域的 GitHub Trending Top 10
→ 我读取 collector.md，按定义执行，保持对话上下文
→ 可以立即看到执行过程和结果
```

**场景 2：需要上下文**
```markdown
你: 我们刚才讨论的那个问题，@analyzer 再分析一下
→ 我知道"刚才讨论的问题"是什么
→ 有上下文连续性
```

### Agent Tool 适用场景

**场景 1：并行搜索**
```markdown
你: 调查项目中所有 API 端点的定义位置
→ 我启动 4 个 Explore Agent 并行搜索：
   - Agent 1 搜索 /routes 目录
   - Agent 2 搜索 /api 目录
   - Agent 3 搜索 /controllers 目录
   - Agent 4 搜索 /handlers 目录
→ 每个独立运行，最后汇总结果
```

**场景 2：复杂独立任务**
```markdown
你: 帮我研究项目中所有与用户认证相关的代码
→ 启动 Explore Agent 进行深度搜索
→ 不占用主对话的上下文窗口
→ 可以搜索大量文件而不影响主对话
```

## AI 知识库项目的使用建议

基于本项目的特点（三 Agent 协作流程）：

| 任务类型 | 推荐方式 | 原因 |
|----------|----------|------|
| **Collector 执行** | @mention | 需要知道当前日期、历史采集状态 |
| **Analyzer 执行** | @mention | 需要访问刚采集的数据 |
| **Organizer 执行** | @mention | 需要知道分析结果和数据状态 |
| **并行采集** | Agent Tool | 同时采集 GitHub + Hacker News + Reddit |
| **代码库探索** | Agent Tool (Explore) | 大量搜索，不占主上下文 |
| **批量分析** | Agent Tool (general-purpose) | 每个 Agent 分析一批，并行加速 |

### 典型工作流

**当前实现（@mention 链式调用）**：
```markdown
你: @collector 采集 GitHub Trending
你: @analyzer 分析采集结果
你: @organizer 整理入库

→ 顺序执行，上下文传递
→ 适合日常使用
```

**优化方案（Agent Tool 并行）**：
```markdown
你: 启动采集流程，并行采集所有数据源

→ Agent 1: @collector (GitHub API)
→ Agent 2: @collector (Hacker News)
→ Agent 3: @collector (Reddit)

→ 合并结果后统一分析
→ 适合批量更新
```

## Agent Tool 的 Subagent 类型

| 类型 | 工具访问 | 适用场景 |
|------|----------|----------|
| **general-purpose** | 全部工具 | 通用任务，代码编写、调试、搜索 |
| **Explore** | 只读（Read/Grep/Glob/WebFetch） | 代码探索、搜索、信息收集 |
| **claude-code-guide** | 文档和帮助 | 询问 Claude Code 功能、配置问题 |
| **Plan** | 所有工具（不可 Edit/Write） | 设计实现方案、架构规划 |

## 性能与成本考虑

### 上下文窗口管理

```markdown
@mention 方式：
- 优点：无额外成本
- 缺点：占用主对话上下文（每次 @mention 都会引用 Agent 定义文档）

Agent Tool 方式：
- 优点：不占主上下文，独立计费
- 缺点：每次调用都是独立的，无法共享上下文
```

### 最佳实践

```markdown
1. 简单任务优先使用 @mention
2. 大量搜索/处理使用 Agent Tool（避免主上下文溢出）
3. 需要结果综合时，让 Agent Tool 返回摘要而非原始数据
4. 并行任务必须使用 Agent Tool
```

## 总结

| 方式 | 一句话总结 |
|------|-----------|
| **@mention** | "我在当前对话中扮演这个角色" |
| **Agent Tool** | "启动一个独立的助手去干活" |

**选择原则**：
- 需要**上下文连续性** → @mention
- 需要**并行加速** → Agent Tool
- 需要**独立处理** → Agent Tool
- 只是**简单调用** → @mention

对本项目来说，日常的三 Agent 工作流使用 **@mention** 更为合适和直观。只有在需要并行采集多个数据源或大规模代码探索时，才考虑使用 **Agent Tool**。
